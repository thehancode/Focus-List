PHASE 1 IMPLEMENTATION BRIEF
Native rendering proof of concept for tui-kanban

Purpose
=======

Create a real native application window that does not need Ghostty or any other
terminal emulator, while preserving the current terminal-grid appearance and
behavior as closely as possible.

This phase is a proof of concept, not a complete GUI rewrite. Keep Ratatui as the
layout engine. Render Ratatui's in-memory cell buffer into an eframe/egui window.
The existing terminal executable must continue to work so it can be used as the
visual and behavioral reference.

Do not change the task JSON schema, storage location, shortcuts, task behavior,
colors, animation durations, or wording in this phase unless a change is strictly
required to compile the native prototype. Do not replace the Ratatui UI with egui
widgets yet.


Target result
=============

The repository should produce two programs:

1. tui-kanban
   The existing terminal version. It must behave exactly as it does now.

2. tui-kanban-native
   A native eframe window. It must not enter terminal raw mode, use the alternate
   screen, or require Ghostty. It should show the existing Ratatui interface in a
   fixed-width character grid and accept the existing keyboard controls.

The native proof of concept should run on Linux first, while keeping all code and
dependencies compatible with Windows and macOS.


Read before making changes
==========================

Important current architecture:

- src/model.rs contains serializable application data.
- src/storage.rs contains JSON loading and saving.
- src/app.rs contains state, behavior, keyboard handling, notices, chords, and
  animation state.
- src/ui.rs renders the complete UI into ratatui::Frame.
- src/main.rs owns the Crossterm terminal event loop.
- The current minimum grid is 50 columns by 12 rows.
- The current loop ticks at approximately 30 FPS (33 ms).
- Task transition animation duration is 220 ms.
- All 28 existing tests pass before this work begins.

Run these commands before editing and save the result for comparison:

    cargo test
    cargo run --release


Development prerequisites
=========================

All platforms
-------------

Install a current stable Rust toolchain with rustup. This repository uses Rust
edition 2024, so do not use an old distribution-provided Rust compiler.

Verify:

    rustup update stable
    rustup default stable
    rustc --version
    cargo --version

The repository currently builds with Rust 1.97.1. Use that version or a newer
stable compiler. Keep Cargo.lock committed because this is an application.

Linux (Ubuntu/Debian development machine)
-----------------------------------------

Install the compiler, pkg-config, X11/Wayland keyboard and graphics development
packages commonly needed by winit/eframe:

    sudo apt update
    sudo apt install build-essential pkg-config libx11-dev libxcursor-dev \
      libxi-dev libxrandr-dev libxkbcommon-dev libwayland-dev \
      libgl1-mesa-dev

If the selected eframe feature set or clipboard integration asks for additional
packages, install the package named in the compiler/pkg-config error. Do not add
GTK, Qt, Electron, Node.js, or a WebView to solve the rendering problem.

Linux (Fedora development machine)
----------------------------------

Use the equivalent packages:

    sudo dnf group install "Development Tools"
    sudo dnf install pkgconf-pkg-config libX11-devel libXcursor-devel \
      libXi-devel libXrandr-devel libxkbcommon-devel \
      wayland-devel mesa-libGL-devel

Windows development machine
---------------------------

Install:

- Rust through rustup using the stable-x86_64-pc-windows-msvc toolchain.
- Microsoft Visual Studio Build Tools with "Desktop development with C++".
- A recent Windows SDK selected by that workload.

Verify in a Developer PowerShell:

    rustup default stable-x86_64-pc-windows-msvc
    cargo test

The native release should use the Windows GUI subsystem so launching the final
native .exe does not also open a console window. Add that attribute only to the
native binary and only under cfg(windows); never add it to the TUI binary.

macOS development machine
-------------------------

Install the Apple command-line development tools and Rust:

    xcode-select --install
    rustup update stable

Apple Silicon uses target aarch64-apple-darwin. Intel uses
x86_64-apple-darwin. macOS binaries must ultimately be built on macOS if they
will be signed, notarized, and placed in a proper .app bundle.


Dependencies
============

Add eframe/egui using a current compatible release. At the time this brief was
written, eframe 0.35 is current. Pin the compatible minor version in Cargo.toml
and let Cargo record the exact transitive versions in Cargo.lock.

For the first prototype, prefer eframe's normal native defaults unless they fail
on the target machine:

    eframe = "0.35"

Do not add Tauri, Dioxus, Slint, Iced, SDL, GTK, Qt, a browser runtime, or a
terminal-emulator library. Do not add an async runtime.

Ratatui remains a normal dependency. The native program may still compile
Crossterm transitively during Phase 1, but it must not initialize or call the
Crossterm terminal backend at runtime.

If binary size or graphics compatibility becomes a demonstrated problem, test
eframe's glow renderer as a separate follow-up. Do not spend Phase 1 changing
rendering backends without an observed problem.


Required source layout changes
==============================

1. Turn the shared code into a library
--------------------------------------

Add src/lib.rs containing public module declarations for:

    pub mod app;
    pub mod model;
    pub mod storage;
    pub mod ui;

Remove the equivalent private `mod` declarations from src/main.rs. Update the
terminal binary to import shared types/functions from the library crate. The
Rust crate path for package `tui-kanban` is `tui_kanban`.

Do not move or substantially rewrite app.rs, model.rs, storage.rs, or ui.rs.
Their existing `crate::...` imports should resolve naturally when compiled as
part of the library.

After this step, immediately run:

    cargo test
    cargo check --bin tui-kanban

All existing tests and the original binary must still pass.

2. Add the native binary
------------------------

Create:

    src/bin/native.rs

Add an explicit Cargo.toml binary entry so its public name is stable:

    [[bin]]
    name = "tui-kanban-native"
    path = "src/bin/native.rs"

On Windows, place this at the top of native.rs:

    #![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]

The native binary should:

- Install a native panic/error reporting strategy without calling
  ratatui::restore().
- Resolve the same existing data directory for Phase 1.
- Call App::load exactly once.
- Start eframe::run_native with an initial window near a 100 x 28 character
  grid, a sensible title, resizable true, and a dark background.
- Own one wrapper struct such as NativeApp containing the existing App plus any
  renderer state and a visible fatal-error string.
- On every eframe update, call App::tick(now), translate input, render the
  Ratatui buffer, and close the viewport when app.should_quit becomes true.

Do not call the existing terminal `run` function from the native program.


Native character-grid renderer
==============================

First implementation strategy
-----------------------------

Use ratatui::backend::TestBackend and ratatui::Terminal as an in-memory renderer.
This is acceptable for the proof of concept and avoids changing ui.rs.

For each frame:

1. Obtain the drawable egui rectangle in logical points.
2. Determine the width and height of one monospace cell.
3. Compute columns = floor(available_width / cell_width).
4. Compute rows = floor(available_height / cell_height).
5. Clamp each value to u16 safely. Never cast a negative or unbounded float.
6. Create or resize a TestBackend to exactly that grid size.
7. Call terminal.draw(|frame| tui_kanban::ui::render(frame, &app)).
8. Read terminal.backend().buffer().
9. Paint the buffer into the egui rectangle.

Recreating a 100 x 28 TestBackend during the prototype is acceptable. Once the
result is correct, it may be retained and resized only when the grid dimensions
change. Correctness is more important than premature optimization.

Do not parse ANSI escape sequences. The Ratatui Buffer already contains the
final symbols and styles; use it directly.

Painting rules
--------------

For every buffer cell:

- Paint its background rectangle first.
- Paint its symbol at the top-left/baseline position for that grid cell.
- Map ratatui::style::Color::Rgb directly to egui::Color32.
- Map Reset background to the existing application background RGB (13, 15, 24).
- Map Reset foreground to the existing text RGB (221, 224, 235).
- Also implement sensible mappings for the named and indexed Ratatui colors so
  the renderer does not panic if the UI later uses them.
- Respect BOLD, ITALIC, DIM, and CROSSED_OUT modifiers as closely as practical.
- A crossed-out cell can be implemented by drawing a horizontal line through
  the glyph if the chosen egui text primitive does not expose strike-through.
- Never panic on an empty symbol, a multi-codepoint symbol, an unknown modifier,
  or a double-width Unicode glyph.

The current UI includes `›`, `◌`, `●`, `✓`, `✦`, `✧`, `·`, `■`, arrows and
rounded box-drawing characters. Verify every one visually. A font that renders
letters but replaces these symbols with boxes is not acceptable.

Start with one draw operation per non-empty cell if that is easiest. After the
result is correct, adjacent cells sharing foreground, background, font style,
and baseline may be combined into runs. Do not optimize by changing UI layout.

Font handling
-------------

Use one stable monospace font for measuring and painting. The cell advance must
not depend on the particular character. Do not use the user's Ghostty font as an
implicit dependency.

For the initial local proof, egui's built-in monospace font plus its fallback
fonts may be used. Before distributing binaries, bundle a redistributable
monospace font with sufficient Unicode coverage. Good font assets must include
their upstream license text in assets/licenses/. Do not commit a font copied
from an operating system or from the user's private font directory.

If a primary monospace font lacks the required geometric symbols, register a
redistributable Unicode symbol fallback. Confirm that fallback glyph advances
still align to the grid; center a fallback glyph inside the cell if necessary.

DPI and resizing
----------------

Use egui logical points, not raw physical pixels. Let eframe handle the OS scale
factor. Measure the font after installing the font definitions and derive the
cell height and monospace advance from that same FontId.

Do not stretch the text when the window size is not an exact multiple of cell
size. Center the grid or place any remainder on the right and bottom. The UI
already shows its "Terminal too small" message below 50 x 12, so pass the real
small grid through instead of enforcing a larger window internally.

Avoid accepting a zero-column or zero-row backend. If the drawable area is too
small even for one cell, paint only the background and wait for a resize.


Input adapter for Phase 1
=========================

App::handle_key currently accepts Crossterm KeyEvent. Phase 2 will remove that
coupling. For this proof of concept, implement a small adapter in native.rs that
converts egui input into Crossterm events.

Required keys:

- Arrow keys, Home, End, Enter, Escape, Tab, Backspace, and Delete.
- Printable text, including Space.
- Ctrl+C.
- Key repeat for navigation and editing.
- The existing Space-then-key chord must preserve its 750 ms timing.

Avoid sending the same printable key twice. egui may provide both a physical key
event and a text event:

- Use key events for navigation/control keys.
- Use text events for printable characters.
- While the prompt editor is open, feed committed text as characters in order.
- Ignore command/control modified text except for explicitly supported commands.

Full IME composition and multi-character paste are Phase 2 concerns because the
current App API only accepts one KeyEvent at a time. It is acceptable for Phase 1
to feed committed/pasted text one Unicode scalar at a time, but it must not
corrupt UTF-8 or slice a string at a non-character boundary.

When q or Ctrl+C makes app.should_quit true, send an egui viewport close command.
Closing the native window through its OS close button should exit normally and
must not manipulate the user's terminal.


Animation and repainting
========================

For the first proof of concept, request another repaint after 16 or 33 ms while
the window is visible. This preserves the existing marquee and 220 ms pulse and
is simple to verify.

Confirm visibly that:

- The selected long title scrolls at the configured speed.
- The background pulse blends from violet toward the new status color.
- Footer sparkle symbols advance during a transition.
- Notices disappear at the existing times.
- The Space chord expires after 750 ms.

Do not introduce a new animation library during Phase 1. Later work should stop
continuous idle repainting and schedule only the next animation, marquee, chord,
notice, or daily-reset deadline.


Error handling
==============

Do not unwrap errors caused by window creation, data loading, rendering, font
loading, or storage. Return an error before the eframe loop when possible. If a
render error happens inside the loop, store it in NativeApp and paint a readable
error panel rather than panicking every frame.

The native program must not install the terminal panic hook from src/main.rs and
must never call ratatui::restore().


Tests and validation
====================

Required automated checks
-------------------------

Run all of the following after implementation:

    cargo fmt --check
    cargo test
    cargo check --all-targets
    cargo build --release --bin tui-kanban
    cargo build --release --bin tui-kanban-native

Add focused unit tests for pure renderer helpers:

- Ratatui RGB, Reset, named, and indexed color conversion.
- Grid-size calculation at normal, fractional-DPI, tiny, and enormous sizes.
- No division by zero for invalid font metrics.
- Modifier conversion where the helper is pure.
- Special-symbol cells do not panic.

Keep the existing Ratatui TestBackend rendering tests unchanged. They are useful
regression coverage for both front ends.

Manual parity checklist
-----------------------

Open the TUI and native binaries side by side using the same JSON data. Verify:

- Header text, counts, and view label.
- Exact palette: background, panel, text, muted, violet, amber, cyan, green, red.
- Rounded borders and all special symbols.
- Selection, Up/Down and j/k.
- Space then f status transition.
- Space then Up/Down reorder.
- New, duplicate, edit, and delete overlays.
- Editor cursor movement, Home/End, Backspace/Delete, Enter, Tab, and Escape.
- Doing focus, completed history, help, and settings.
- Marquee and wrapped-title setting.
- Daily completion strip.
- Small-window fallback below 50 x 12.
- Repeated resizing at 100%, 125%, 150%, and 200% OS scaling if available.
- Idle and animated CPU usage, with no runaway error loop.
- The original TUI remains unchanged.

Phase 1 is complete only when the native binary can be launched directly from a
desktop/file manager and used for the normal task workflow without Ghostty.


What is explicitly out of scope
===============================

Do not perform these changes in Phase 1:

- Rewriting panels, rows, or modals as normal egui widgets.
- Changing the JSON schema or data path.
- Adding drag-and-drop or mouse-first behavior.
- Replacing Crossterm inside App with a platform-independent input enum.
- Adding a new audio library. The terminal bell may be temporarily silent in the
  native prototype; do not write ASCII BEL from the native app.
- Creating installers, auto-update, cloud sync, accounts, telemetry, or a web UI.
- Removing the TUI binary.
- Adding a background service or async runtime.

Record native sound and the platform-independent input refactor as Phase 2 work.


Source-code and binary distribution
===================================

License and repository hygiene
------------------------------

Cargo.toml declares MIT, but this repository currently has no tracked LICENSE
file. Before any public source or binary release:

1. Add the full MIT license as LICENSE with the correct copyright holder/year.
2. Add license files for every bundled font or other asset under
   assets/licenses/.
3. Mention bundled assets and their licenses in README.md or a NOTICE file.
4. Keep Cargo.lock in version control for reproducible application builds.
5. Never distribute target/, local tasklists/, user JSON data, editor files,
   signing keys, or platform signing credentials.

The existing .gitignore already excludes target/ and tasklists/. Verify with:

    git status --short
    git ls-files

Do not include PROMPTS.md in a release archive unless the owner intentionally
wants the build history/prompts to be public.

Distributing the source
-----------------------

The preferred source distribution is a public Git repository plus immutable
version tags. For each release:

1. Update the version in Cargo.toml.
2. Run formatting, tests, and release builds.
3. Commit Cargo.lock.
4. Create a signed or annotated tag such as v0.2.0-native-preview.1.
5. Attach a source archive generated from the tag, not from the working tree.

A source archive should contain at least:

    Cargo.toml
    Cargo.lock
    LICENSE
    README.md
    src/
    assets/                 if fonts/icons are bundled
    assets/licenses/

People should be able to build it with:

    cargo build --release --bin tui-kanban-native

If source is public under MIT, recipients may use, modify, and redistribute it
subject to retaining the MIT copyright and permission notice. Bundled fonts keep
their own licenses; the application MIT license does not replace those licenses.

Distributing preview binaries
-----------------------------

Phase 1 does not require installers. Publish compressed portable previews built
on each native operating system:

- Windows x86-64: tui-kanban-native.exe in a .zip.
- Linux x86-64: tui-kanban-native in a .tar.gz. State the minimum supported
  distribution/glibc expectation; an AppImage can be added later.
- macOS Apple Silicon: a proper .app bundle in a .zip or DMG. Unsigned builds are
  suitable only for private testing because Gatekeeper will warn or block users.
- macOS Intel can be added if there is actual demand.

Do not claim that a Linux-built Windows or macOS executable is release-tested.
Use a CI matrix or native machines for each OS. At minimum, CI should run tests
and build the native binary on ubuntu-latest, windows-latest, and macos-latest.

For public production distribution after the proof of concept:

- Sign the Windows executable/installer to reduce SmartScreen warnings.
- Sign and notarize the macOS .app and DMG with an Apple Developer identity.
- Consider AppImage plus .deb for Linux.
- Generate SHA-256 checksums for every artifact.
- Publish the matching source tag/archive with every binary release.
- Document where each OS stores task data before changing to platform-native
  directories in Phase 2.

Never place code-signing certificates or private keys in the repository. Store
them in the release service's encrypted secrets and restrict release workflows
to protected tags.


Implementation handoff report
=============================

When the coding work is finished, report:

- Every file added or changed.
- The exact eframe version and renderer in use.
- Whether a font was bundled and its license.
- Which operating systems were actually compiled and manually tested.
- Results of cargo fmt, cargo test, cargo check, and both release builds.
- Known visual differences from Ghostty.
- Known input limitations, especially paste and IME.
- Release artifact sizes.
- Any Phase 2 follow-ups discovered during implementation.

Do not describe Phase 1 as complete if only a blank eframe window or static
screenshot works. It must render live App state, accept the principal shortcuts,
persist tasks through the existing store, resize, and animate without Ghostty.
