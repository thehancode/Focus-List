FOCUS LIST: CURRENT PROGRAM AND FEATURE SPECIFICATION
=====================================================

Purpose of this document
------------------------

This is the behavioral contract for migrating the Rust application to Flutter.
It describes the active program as implemented in src/app.rs, src/model.rs,
src/storage.rs, src/ui.rs, src/main.rs, and src/bin/native.rs. The Rust tests are
also treated as executable documentation. A Flutter implementation should keep
these rules unless a migration task explicitly changes them.


1. PRODUCT SUMMARY
------------------

Focus List is a small, keyboard-first task manager. It organizes tasks into
named lists and three statuses: Pending, Doing, and Done. It favors a focused,
single-column workflow rather than a conventional three-column kanban board.

The active interfaces are:

* A terminal UI built with Ratatui/Crossterm.
* A Linux-style native desktop preview built with eframe/egui. It renders the
  same Ratatui character-grid UI and uses the same application state and files.

The Flutter migration is expected to replace both interfaces. Terminal support
does not need to be retained.


2. DATA MODEL
-------------

2.1 Task status

Status has exactly three values serialized as lowercase strings:

* pending
* doing
* done

The forward status cycle is:

    Pending -> Doing -> Done -> Pending

Normal list display order is Doing, Pending, Done. This differs from the enum's
general ALL order (Pending, Doing, Done), which is not used by the active list
view.

2.2 Task

Each task contains:

* id: UUID.
* title: non-empty text.
* status: pending, doing, or done.
* created_at: UTC timestamp.
* updated_at: UTC timestamp.
* completed_at: optional UTC timestamp, populated while a non-recurring or
  recurring task is currently Done.
* daily: boolean, false by default.
* completion_history: UTC timestamps for every recorded completion of a daily
  task.

A newly created task receives a new UUID, Pending status, identical created and
updated timestamps, no completion time, daily=false, and empty history.

2.3 Task list

Each list contains:

* schema_version: currently 1.
* id: UUID.
* name: non-empty and unique among loaded lists, ignoring case.
* created_at: UTC timestamp.
* tasks: one ordered array of Task objects.

The first list is created automatically when no lists exist and is named
"Tasks". At least one list must always exist.

2.4 Ordering invariant

The tasks array is the canonical order. Views filter it without independently
sorting it. Therefore:

* Tasks with a given status appear in their relative array order.
* Advancing a status changes the status but does not relocate the task in the
  array.
* Reordering swaps the selected task with the previous/next task having the
  same status. It does not swap with a task in another status.
* List order is creation/load order. Files are loaded by created_at, then UUID,
  with migration-path details used only as a final tie breaker.

This invariant must remain explicit in Flutter and in a future backend schema.
If the backend later normalizes tasks into rows, it will need an order/rank
field that reproduces this per-list, per-status order.


3. PERSISTENCE AND STARTUP
--------------------------

3.1 Linux paths

The Rust app uses:

    $XDG_DATA_HOME/tui-kanban/tasklists/

or, if XDG_DATA_HOME is absent:

    $HOME/.local/share/tui-kanban/tasklists/

Each task list is stored in its own pretty-printed JSON file named:

    <task-list-uuid>.json

Settings are stored below the same tasklists root at:

    config/settings.json

3.2 Task-list JSON shape

The persisted shape is equivalent to:

    {
      "schema_version": 1,
      "id": "UUID",
      "name": "Tasks",
      "created_at": "UTC RFC-3339 timestamp",
      "tasks": [
        {
          "id": "UUID",
          "title": "Example",
          "status": "pending",
          "created_at": "UTC RFC-3339 timestamp",
          "updated_at": "UTC RFC-3339 timestamp",
          "completed_at": null or "UTC RFC-3339 timestamp",
          "daily": false,
          "completion_history": []
        }
      ]
    }

Serde omits completed_at when null, daily when false, and completion_history
when empty. A compatible reader must accept both omitted and explicit default
values.

3.3 Settings

The persisted settings and defaults are:

    {
      "marquee_speed_ms": 180,
      "long_title_display": "marquee",
      "native_font_size": 16
    }

Validation ranges:

* Marquee speed: 50 through 1000 milliseconds; adjusted in steps of 25 ms.
  Lower values scroll faster.
* Long-title display: marquee or wrap.
* Native font size: 10 through 28 points; adjusted in steps of 1.

3.4 Save behavior

Mutations are saved immediately. Linux writes use a temporary .json.tmp file
followed by rename, so replacing a list or settings file is atomic on the same
filesystem. Deleting a list deletes only its UUID file.

3.5 Load, validation, and migration behavior

At startup the app:

1. Creates the task-list directory if needed.
2. Reads every .json file directly in that directory.
3. Skips malformed or invalid list files and shows a timed error notice. It
   reports the first error plus the count of additional file errors.
4. Rejects unsupported schema versions, blank list names, and blank task titles.
5. Migrates older arbitrarily named JSON list files (for example tasklist.json)
   to UUID filenames without merging their tasks.
6. Removes an identical duplicate legacy file when a canonical UUID file for
   the same list already exists.
7. Skips conflicting duplicate UUIDs and reports a warning.
8. Disambiguates duplicate names found during loading by adding " (2)", " (3)",
   and so on, comparing names case-insensitively, then saves the corrected list.
9. Creates the default empty "Tasks" list if no valid lists remain.
10. Loads settings, creating the default settings file if absent.
11. Opens List view, selects the first visible task, and starts with sound on.


4. APPLICATION STATE AND VIEW MODES
-----------------------------------

The app keeps a current list index, a selected task UUID, a view mode, an
optional modal overlay, transient animation/notice state, and Space-chord state.

Task selection is by UUID rather than row index. Up/down movement clamps at the
first and last visible task; it does not wrap. If a view has no visible tasks,
selection is null and movement does nothing.

4.1 List view

This is the normal per-list view. It shows three vertical sections in order:

1. Doing
2. Pending
3. Done

Every section has a colored icon, uppercase label, count, and either its tasks
or an empty message. The panel scrolls as necessary to keep selection visible.

4.2 Doing Focus view

This view shows only Doing tasks in the current list. Pressing c toggles between
Focus and List, but Focus can only open if the current list has a Doing task.
Otherwise the app stays in its current view and shows "No Doing tasks to focus".

Moving a Pending task to Doing automatically opens Focus view. Moving a Doing
task to Done returns to List view if no Doing tasks remain in the current list.
If other Doing tasks remain, Focus stays open when it was already open.

4.3 Completed view

This is per-list completion history, toggled with v. Entries are ordered newest
completion first and show a checkmark, struck-through title, and local
YYYY-MM-DD HH:MM timestamp.

For a non-daily task, its current completed_at produces one history entry. For
a daily task, every completion_history timestamp produces an entry, even after
the task has reset to Pending. Thus the same daily task may appear many times.

The r command reverts a selected Done task to Doing, clears completed_at,
updates updated_at, and opens Focus. It does not delete existing daily history.

4.4 Multi view

Ctrl+A toggles Multi view. It shows Doing and Pending work across every list:

* Lists appear in list order.
* A list is omitted if it has neither Doing nor Pending tasks.
* Each included list has an uppercase list heading.
* Within each list, Doing is shown first and Pending second.
* Empty status sections are omitted.
* Done tasks never appear.
* If nothing is visible, an all-lists empty message is shown.

No tab is visually selected while Multi view is open. Clicking any tab chooses
that list and returns to its normal List view.

Up/down selection crosses status and list boundaries. As selection crosses into
another list, that list becomes the internal current list. Consequently create,
edit, delete, status, and reorder actions apply to the selected task's owning
list. Creating a task with no selected task uses whichever list is internally
current.

Multi view status workflow:

* Pending -> Doing opens that list's Focus view and remembers that Focus was
  entered from Multi.
* Doing -> Done then returns to Multi and selects the first remaining visible
  Multi task.
* Advancing an already-Doing task directly from Multi to Done also remains or
  returns to Multi.
* Ctrl+A from Multi returns to the normal List view of the internally current
  list.


5. TASK OPERATIONS
------------------

5.1 Create

Press n to open a new-task editor. The new task starts non-daily. Enter saves
and Esc cancels. Whitespace is normalized on save: runs of whitespace,
including line breaks, become single spaces and surrounding whitespace is
removed. Empty normalized titles are rejected with an error notice. The new
task is appended to the current list's tasks array, selected, and saved.

5.2 Edit

Press e with a selected task to edit it. The editor is prefilled with the title
and daily flag. Saving updates title, daily, and updated_at but preserves UUID,
status, created_at, completed_at, and completion history.

5.3 Duplicate

Press d with a selected task. This opens the new-task editor prefilled with the
selected title and daily flag. Saving creates a genuinely new Pending task with
a new UUID and timestamps; status and completion history are not copied.

5.4 Delete

Press x with a selected task to open a destructive confirmation. y/Y confirms;
n/N or Esc cancels. Confirming removes the task, saves the owning list, closes
the modal, and selects the first task visible in the current view.

5.5 Advance status

Press and hold/tap Space, then press f/F while the chord is armed. This advances
one step through Pending -> Doing -> Done -> Pending and updates updated_at.

On entry to Done:

* completed_at becomes the same value as updated_at.
* A daily task also appends that timestamp to completion_history.

On entry to Pending:

* completed_at is cleared.
* For a daily task, completion-history entries from the current local calendar
  day are removed. Older history is retained.

The transition is saved immediately and produces a 220 ms colored pulse/status
animation. In the terminal frontend, it emits ASCII BEL if sound is enabled.
The native preview intentionally suppresses terminal BEL and has no replacement
native sound yet.

5.6 Grab and reorder

Space arms a 750 ms chord. While armed, Up/k or Down/j moves the selected task
one position among tasks having the same status. Each successful direction key
refreshes the 750 ms deadline, allowing repeated movement over an arbitrary
distance. Repeated Space key events also refresh the armed time. Releasing Space
immediately disarms grab mode. A direction key after release only changes
selection. Reorder is unavailable in Completed view and clamps at group ends.

5.7 Daily tasks

Tab inside the task create/edit dialog toggles daily yes/no. On every app tick,
daily tasks in the current list that are Doing or Done and whose updated_at is
from an earlier local calendar date reset to Pending. Reset clears completed_at,
sets updated_at to now, preserves completion_history, saves, and reselects the
first visible task.

When a selected task is daily, the footer's top line shows recent activity. The
leftmost cell is today, then yesterday, and so on. A filled square means at least
one completion on that local date and a dot means none. This activity line takes
display priority over notices and transition/chord feedback.


6. TASK-LIST OPERATIONS AND TABS
--------------------------------

6.1 Create list

Ctrl+N opens a list-name dialog. The name is whitespace-normalized. Blank names
or names matching any existing list case-insensitively are rejected. Saving
creates a new empty UUID list, appends it, switches to it, returns to List view,
and shows a success notice.

6.2 Rename list

F2 or Ctrl+R opens a dialog prefilled with the current name. The same blank and
case-insensitive uniqueness rules apply, excluding the list being renamed.
Saving persists immediately. A failed save restores the old in-memory name.

6.3 Delete list

Ctrl+X opens confirmation when two or more lists exist. The confirmation names
the list and warns that all its tasks will be lost. y/Y deletes; n/N or Esc
cancels. The final remaining list cannot be deleted; the app instead shows an
error notice. After deletion the app selects the list now occupying the old
index, or the preceding list if the deleted list was last, and resets to List.

6.4 Switch and click tabs

Tab cycles forward and Shift+Tab cycles backward, wrapping at both ends. List
switching resets to List view, clears chord/animation/Multi-return state, and
selects the first visible task.

The second character row is a tab bar. The current list is highlighted except
in Multi view. When all labels do not fit, it keeps the current tab visible and
uses left/right chevrons to indicate hidden tabs. In the native desktop preview,
clicking a visible tab selects it. The chevrons themselves are indicators, not
buttons.


7. TEXT EDITING AND MODALS
--------------------------

While a modal is open, normal application shortcuts and task actions are
blocked; only that modal's handler receives keys.

Task-title editor:

* Supports Unicode insertion at the cursor.
* Left/Right move by Unicode scalar boundary.
* Home/End move to the start/end of the current manual line.
* Up/Down retain the character column as closely as possible between manual
  lines.
* Backspace/Delete remove the previous/next character.
* Tab toggles daily status.
* Enter saves and Esc cancels.
* The editor visually wraps long content and shows a block cursor.

List-name editor:

* Supports insertion, Left/Right, Backspace/Delete.
* Home/Up go to the start; End/Down go to the end.
* Enter saves and Esc cancels.

Delete confirmations accept y/Y, n/N, and Esc as described above.


8. SETTINGS, HELP, NOTICES, AND FEEDBACK
----------------------------------------

Press g to open Settings. Up/Down or j/k selects one of three rows. Left/Right
or h/l changes the value. Esc, g, or q closes Settings. Every setting change is
persisted immediately.

Settings rows:

1. Marquee speed, +/- 25 ms, clamped to 50..1000 ms.
2. Long titles, toggled between Marquee and Wrap.
3. Native font size, +/- 1 pt, clamped to 10..28 pt.

Press s to toggle transition sound for the current process. It starts enabled
but is not persisted in settings.

Press ? to open Help. Esc, ?, or q closes it. Help lists the major shortcuts.

Success notices last 2 seconds; errors last 6 seconds. Startup file warnings
last 8 seconds. Notices appear in the footer and are truncated to available
width. Footer feedback priority is daily activity, notice, transition animation,
Space-armed hint, then blank. A second footer line continuously shows compact
shortcut hints.


9. VISUAL AND INTERACTION DESIGN
--------------------------------

The interface uses a dark background and panel, light text, muted secondary
text, and semantic accents:

* Violet: selection, tabs, and primary chrome.
* Amber: Pending, grab hint, and Multi panel border.
* Cyan: Doing and Focus border.
* Green: Done, completed history, and successful feedback.
* Red: errors and destructive dialogs.

Status icons are ◌ Pending, ● Doing, and ✓ Done. The selected task uses a violet
background, dark text, bold weight, and a leading › marker. Unselected tasks use
a leading dash. Done tasks are muted and struck through.

Long-title modes:

* Marquee: only the selected overflowing title scrolls. Manual newlines become
  spaces. The loop has a "   •   " separator and advances at the configured
  interval. Unselected overflowing titles are truncated with an ellipsis.
* Wrap: titles, including manual line breaks, render on multiple character-grid
  rows. Each row remains clickable in the native preview.

The layout requires at least a 30 x 20 character grid. Smaller areas show only
a resize message. The header names the product and current view, row two holds
tabs, the central bordered panel scrolls, and the final two rows are the footer.

The native preview bundles regular and bold UbuntuMono Nerd Font Mono. It opens
at an approximately 35-column by 100-row grid, is borderless, can be dragged by
the top row, resized from its left/right/bottom edges, and repaints at about 30
frames per second for animations and marquee motion.


10. KEYBOARD REFERENCE
----------------------

Global when no modal is open:

* Up/Down or j/k: move selection, clamped at view boundaries.
* Space then f/F: advance selected task status.
* Space then Up/Down or j/k: reorder selected task within its status.
* Space release: end grab/reorder mode.
* n: create a task.
* e: edit selected task.
* d: duplicate selected task through the new-task editor.
* x: confirm deletion of selected task.
* r: revert selected Done task to Doing.
* c: toggle Doing Focus when Doing work exists.
* v: toggle Completed view.
* Ctrl+A: toggle Multi view.
* Tab/Shift+Tab: next/previous task list.
* Ctrl+N: create task list.
* F2 or Ctrl+R: rename current task list.
* Ctrl+X: confirm deletion of current task list.
* g: settings.
* s: sound on/off.
* ?: help.
* q or Ctrl+C: quit.

The terminal handler also accepts raw ASCII control-character encodings for
Ctrl+A, Ctrl+N, Ctrl+R, and Ctrl+X. Flutter does not need that compatibility
layer, but must implement logical shortcuts consistently on desktop and web
where the browser/OS delivers them.


11. POINTER SUPPORT IN THE CURRENT NATIVE APP
---------------------------------------------

Pointer support is intentionally limited:

* Primary-click a visible task row (including a wrapped continuation row) to
  select that task.
* Primary-click a visible list tab to select the list and return to List view.
* Hover over tabs shows a pointing-hand cursor.
* The top character row drags the borderless window.
* Left, right, and bottom edges/corners resize the window with matching cursors.

There is currently no pointer button for create/edit/delete/status changes and
no pointer drag-and-drop task reordering. Those are reasonable Flutter/mobile
enhancements, but they are not existing behavior.


12. NON-FEATURES AND MIGRATION BOUNDARIES
-----------------------------------------

The source contains unused render helpers for a three-column kanban and a
two-month daily-activity popup. They are not reachable in the active app and
must not be treated as required current features.

The current app has no account, network sync, backend, collaboration, due dates,
priorities, tags, subtasks, attachments, audio notes, search, or notifications.
The Flutter architecture should leave clean extension points for backend sync
and audio notes without inventing those features in migration version 1.
